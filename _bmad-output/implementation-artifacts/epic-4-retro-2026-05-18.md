## What Hurt

### Recurring Friction

1. **Client-selection asymmetry** (4-1, 4-3, 4-5). Admin client for reads, customer client for writes — but it's nowhere documented at the project level. Every Server Action had to re-derive "which client and why" inline.
2. **JSONB `as Type` casts without runtime shape validation** (4-1, 4-3, 4-5). `as VariantGroup[]`, `as AvailabilitySchedule`, `as { restaurant_id, table_number }` — all unchecked. Today's bug exposed the diagnostic cost: when `app_metadata` is missing, we silently fall through to `restaurant_id = undefined` with no clear error.
3. **ARIA structure + focus management** (4-3, 4-4, 4-5). Two review cycles to get CartBar right (`role="complementary"` on a clickable `<div>` was not focusable / `aria-live` inside an interactive element is suppressed). Required architectural refactors in 4-4.

### Late-Discovered Bug — The Headline Lesson

**Issue:** After Epic 4 was marked complete with all 257 unit tests green, real-app testing revealed that `submitOrder` consistently failed with error 42501 ("new row violates row-level security policy for table 'orders'").

**Diagnostic Path (4 rounds):**
1. First guess: stale JWT from before custom_access_token_hook registration. → JWT decoded; claims were correct.
2. Second guess: hook not registered in Dashboard. → Verified registered.
3. Third guess: `app_metadata` shape mismatch between `user.app_metadata` (DB record) and `auth.jwt() -> 'app_metadata'` (JWT claims). → Both matched.
4. Root cause: Simulated the INSERT under `SET LOCAL ROLE authenticated` + matching JWT via Supabase MCP `execute_sql`. Confirmed: the INSERT itself was allowed; the `RETURNING id` was rejected because PostgreSQL evaluates **SELECT RLS on the returned row**. The only SELECT policy on `orders` is `owner_select_orders` (requires `get_my_restaurant_id()` — null for anonymous customers).

**Root Cause:** Customer-facing `INSERT ... RETURNING id` requires customer-readable SELECT RLS. None existed. PostgreSQL surfaced the failure with the same error code as a WITH CHECK violation — that's what made diagnosis take 4 rounds.

**Fix Applied (today):** Dropped `.select('id').single()` from the `submitOrder` insert. Removed `orderId` from `SubmitOrderData` (the confirmation screen never displayed it). The customer's INSERT now goes through cleanly; the `customer_insert_order` WITH CHECK still validates JWT claims against the new row, so the security guarantee is preserved. Tests updated; 257/257 still pass.

## Key Lessons (Captured for the Project Record)

1. **Mocked unit tests are not a substitute for an integration path through real RLS.** For customer-facing Server Actions that hit Postgres with RLS, "done" must include at least one real INSERT against the live (dev) database — either via e2e/Playwright or a manual smoke step in the story DoD.

2. **Postgres error 42501 is overloaded.** Identical error code covers both WITH CHECK failures *and* RETURNING-SELECT failures. When debugging RLS, isolate by trying the INSERT **without `RETURNING`** first.

3. **Customer-facing INSERTs default to no `.select()`.** Only add `RETURNING` when (a) the client genuinely consumes the returned row, and (b) a customer-readable SELECT policy exists.

4. **Client-selection asymmetry is project lore that should be infrastructure.** Admin client for reads (menu data, RLS-bypassed safely); customer client for writes (RLS validates the JWT). This needs documentation at the project level, not per-story dev notes.

5. **`as` casts on JSONB and `app_metadata` defer pain.** Runtime shape validation pays back in diagnostic clarity, especially under error conditions.

## Significant Discovery Check

**Does Epic 4 require changes to Epic 5 plans?** No fundamental changes.

- Epic 5's owner-side flow doesn't need `orderId` returned to the customer (Realtime delivers the full row with `id`), so today's fix is fully forward-compatible.
- `owner_select_orders` and `owner_update_orders` policies are already in place and were not affected by today's change.
- The lessons captured here apply *during* Epic 5 stories — they don't require replanning the epic.

## Postscript (2026-05-18, evening)

The sprint-change-proposal `sprint-change-proposal-2026-05-18b.md` (approved same day) removes the anonymous customer auth pattern entirely. Customer flow is now sessionless — the URL `(slug, table_number)` is validated by the order-submission Server Action against the database, and the order is inserted via the admin client. No `auth.users` rows are created for customers.

**Impact on this retrospective's headline lesson:**

> *"Mocked unit tests are not a substitute for an integration path through real RLS."*

The lesson **remains fully valid for owner-side Server Actions** (e.g., Epic 5's upcoming `markOrderHandled`, which still goes through `owner_update_orders` RLS). It is **moot for customer-side write paths**, which no longer traverse RLS at all — they validate at the Server Action layer instead. The standing DoD requirement from this retro should now be read as: "any Server Action that touches Postgres under RLS needs a real-DB smoke test before being marked done."

**Impact on the Postgres 42501 / RETURNING-SELECT lesson:** unchanged. Still load-bearing, still in `docs/conventions/supabase-clients.md`.

**Impact on the 17 stale anonymous users** counted in the postscript investigation: purged in the same migration that removed the trigger landmine (`20260518230000_remove_anonymous_customer_pattern.sql`).
